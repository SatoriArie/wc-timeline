# Фракции: полноценная страница вместо модалки — дизайн

Дата: 2026-07-01

## Контекст

Вкладка «Фракции» (`OrganizationsPage` + `OrganizationModal`) сейчас показывает организации
(`Organization` в `public/data/organizations.json`) плитками без гербов (буква-аватар вместо
иконки), а клик открывает модалку с авто-вычисляемым «Составом» (по совпадению
`character.affiliations` со строкой `org.name`) и «Зонами влияния» (по совпадению
`zone.factions`).

Владелец хочет поднять фракции до того же уровня проработки, что и персонажи: реальные гербы
на превью и внутри карточки, явных «Глав(ы)» и «Состав» с учётом бывших участников (со
ссылками на персонажей), и полноценную страницу вместо модального окна — по образцу того, как
уже сделаны персонажи (портрет+краткие детали слева, текст справа), но с собственной навигацией
«назад к списку» вместо оверлея.

«Территории влияния/присутствия» как отдельный курируемый список зон явно вынесены владельцем
в будущее — в этом заходе не делаем, оставляем нынешний авто-блок «Зоны влияния» как есть.

## Решения (согласованы с владельцем)

1. **Формат экрана.** Отдельная полноценная страница (не модалка). Открывается на месте списка
   фракций; есть кнопка «Назад к фракциям».
2. **Модель состава.** Полностью ручной список на самой фракции (не авто-подбор по
   `affiliations`). У каждой записи — персонаж + флаг «бывший».
3. **Территории.** Не делаем в этот заход. Авто-блок «Зоны влияния» (по `zone.factions`)
   остаётся без изменений.

## Изменения данных (`src/data/types.ts`)

```ts
export interface OrgMember {
  id: string;       // id персонажа
  former?: boolean; // true = бывший
}

export interface Organization {
  id: string;
  name: string;
  category: string;
  domain: string;
  note: string;
  color: string;
  emblem?: string;      // путь/URL герба — как character.portrait
  leaders: OrgMember[]; // ручной список, курируется на фракции
  members: OrgMember[]; // ручной список, курируется на фракции
}
```

- `leaders`/`members` — новые обязательные поля типа (пустой массив по умолчанию для новых
  записей); `normOrg()` в `src/data/source.ts:104` проставляет `leaders: o.leaders ?? []`,
  `members: o.members ?? []` для обратной совместимости со старыми записями без этих полей.
- `character.affiliations` (плоский список строк) не меняется и продолжает работать как раньше
  — это отдельный механизм (клик по тегу принадлежности на карточке персонажа → переход на
  фракцию), независимый от нового ручного списка `members` на фракции.
- «Зоны влияния» не меняются — остаются производными от `zone.factions.includes(org.name)`.

## Герб фракции

- Поле `Organization.emblem?: string` — путь к изображению, как `character.portrait`.
- Файлы кладутся в `public/images/emblems/<id>.<ext>`.
- `scripts/wire-images.mjs` расширяется по существующему паттерну (`groupById('portraits')` /
  `groupById('events')` / `groupById('zones')`): добавляется `groupById('emblems')` →
  проставление `org.emblem` в `organizations.json`, с тем же предупреждением о файлах без
  совпадения id.
- Если `emblem` пуст — фолбэк на нынешнюю букву-аватар (без регрессии для ещё не
  проиллюстрированных фракций).

## Карточки-превью (`OrganizationsPage`)

Замена `<span className="member-glyph">{letter}</span>` на изображение герба
(`assetUrl(o.emblem)`), с тем же fallback на букву при отсутствии картinки — по аналогии с
`character-portrait`/`portrait-fallback` в `CharactersPage`.

## Страница фракции (новый компонент `FactionPage`, заменяет `OrganizationModal` как основной
режим просмотра)

Раскладка — по мотивам `CharacterModal` (`char-aside` слева, текст справа), но как отдельная
страница, а не модалка:

- **Шапка**: кнопка «← Назад к фракциям», название, категория/эпитет.
- **Левая колонка**: герб (крупно, `assetUrl(org.emblem)`, фолбэк — буква), под ним краткий
  список деталей (категория, тип/эпитет, число участников состава, число зон влияния).
- **Правая колонка (текст)**:
  - «О фракции» (`note`).
  - «Глава(ы)» — `leaders`, каждая запись кликабельна → модалка персонажа; у бывших — пометка
    «бывший» (как race-transform стрелка/бейдж у персонажа).
  - «Состав» — `members`, тот же паттерн отображения (текущие/бывшие).
  - «Зоны влияния» — как сейчас, клик → модалка зоны.
  - «Летопись фракции» — как сейчас, группировка по эпохам.

### Навигация и стейт в `App.tsx`

- `activeOrg` остаётся state-флагом «какая фракция открыта», но теперь управляет рендером
  основной области, а не модалкой: `page === 'organizations' && activeOrg` → `<FactionPage/>`
  вместо `<OrganizationsPage/>`; `page === 'organizations' && !activeOrg` → список как сейчас.
- `<OrganizationModal/>` убирается из оверлейных модалок в конце `App.tsx`.
- Клик по персонажу/зоне **внутри** `FactionPage` открывает `CharacterModal`/`ZoneModal` как
  оверлей поверх страницы фракции (эти два остаются модалками); `activeOrg` при этом **не**
  сбрасывается — `openOrgChar`/`openOrgZone` в `App.tsx` перестают делать
  `setActiveOrg(null)` перед открытием, чтобы закрытие модалки персонажа/зоны возвращало на
  страницу фракции.
- Диплинк `#/faction/<id>` и хэш-роутинг не меняются по механике — просто на выходе теперь
  полноценная страница вместо модалки.

## Редактирование (`EditModal.tsx`)

Форма типа `organizations` получает:
- Поле «Герб (URL)» — как `Field` «Портрет (URL)» у персонажа.
- Два редактора-пикера персонажей по образцу `RelationsEditor`, но без `kind`: выбор персонажа
  из `allCharacters` + чекбокс «Бывший» + кнопка «Удалить»/«+ Добавить». Один экземпляр для
  `leaders`, второй — для `members`.

## Вне рамок этого спека

- Явный курируемый список «Территорий влияния/присутствия» (зоны) на фракции — отложено,
  зафиксировано как план на будущее. Текущий авто-блок «Зоны влияния» остаётся без изменений.
